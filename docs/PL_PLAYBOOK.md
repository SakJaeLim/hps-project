# PL 운영 플레이북 (4주·4인) — 무엇을 더 할 것인가

지금까지 "무엇을 만들지"(역할·RACI·계약·SDD)는 세웠다. PL의 남은 일은 **사람·의존성·흐름**을 굴리는 것이다.

## A. 이번 주 (가장 큰 레버리지)
1. **킥오프 + 계약 동결 세션 (1회, 전원)** — 아키텍처 기준도(현재 확정본) 공유하고, 4개 계약
   `YardState · CandidatePlan · Violation · Recommendation` 필드를 **같이 확정·동결**한다.
   이 동결이 끝나야 각자 mock으로 병렬개발을 시작할 수 있다(유일한 동기 지점).
2. **긴 리드타임 의존성 즉시 착수** (안 하면 남이 막힌다):
   - 원우3 → **RL 모델 체크포인트 + env 사양(관측/행동/보상) 인수인계**, 선박제원 전사
   - **GPU/Runpod**(LoRA용), **Neo4j 인스턴스**, GitHub 레포·접근 권한
   - 실데이터는 비동기 — 어댑터로 격리(도착 시 교체)
3. **Day-1 "그린"** — 전원 레포 클론 → `pytest`(2 passed/14 xfailed 확인) → Neo4j 기동 → 환경 동일화.
   "내 PC에선 됨" 낭비를 1일차에 제거. (Definition-of-Ready)

## B. 운영 리듬 (가볍게)
- 스탠드업 주 2–3회(15분) + **주 1회 통합 데모**(M1·M2·M3). 소통 채널 1개로.
- **통합은 PL이 자주** — 브랜치 오래 안 벌리고 수직 슬라이스를 항상 작동 상태로(빅뱅 통합 금지).
- 결정은 `ADR` + `CHALLENGE-LOG`에 공개. 팀이 주장을 제기하도록 초대(반박 프로토콜).
- DoD = 테스트 Green + PR 1인 리뷰.

## C. 사람 관리 포인트
- **원우3(도메인)을 저마찰로 생산적이게**: 규칙표·instruction 데이터셋(JSON)·평가 시나리오 **템플릿**을 PL이 먼저 제공.
  Git 마찰로 사이드라인되지 않게 이슈/문서 기여 경로 마련. — RL·규칙·평가라 **임계경로**다.
- **기대치 관리**: 목표는 얇은 슬라이스(상용 능가 아님). M1 수직슬라이스 띄우면 축하. "막혔다" 말해도 안전하게.
- 새 기술(Neo4j·LangGraph)은 workshop 노트북 + `specs/`로 안내, 어려운 부분은 **페어링**.
- 발표·시연(원우1) **일찍** 준비 — 내러티브 리허설.

## D. 킥오프 아젠다 (예: 90분)
1. 기준도·흐름 5분 설명(PL) → 2. 각자 담당·1개월 범위 확인(6.1) → 3. **계약 4개 필드 합의·동결** →
4. Day-1 setup 같이 → 5. 이번 주 의존성 담당·기한 지정 → 6. 채널·리듬 합의 → 7. 첫 TASK 배정(T01–T05·T13).

## E. 막히면 되돌아볼 것
- 누가 "무엇을 기다리고" 있나? → 그 의존성이 PL의 1순위.
- 수직 슬라이스가 이번 주에 도는가? → 안 돌면 범위를 더 깎는다.
- 결정이 PL 머릿속에만 있나? → ADR/CHALLENGE-LOG로 꺼낸다.
