# ADR-0001 · 적재 최적화 엔진 — RL 채택, CP는 조건부 향후 비교

- 상태: 채택(Accepted) · 2026-06
- 관련: specs/00_system_spec.md · specs/02_agent_spec.md · src/snct/engine/

## 맥락
- 도메인 전문가(원우)가 **논문 투고용으로 이미 개발·학습한 RL 적재 모델**이 존재한다.
- 프로젝트 1차 목표는 **"현재 기술의 통합·활용(integration)"** 이며, 최적화 성능의 SOTA 경신이 아니다.
- 외부 검증(2026-06): 적재계획 RL은 연구 단계이고 복잡도↑ 시 성능 저하가 보고됨. 반면 제약계획법(CP/IP)은
  실제 베이 인스턴스 90%를 1초 내 최적해로 푸는 검증된 방법. → 단, 본 프로젝트는 성능 경신이 목표가 아님.

## 결정
1. MVP 적재 최적화 엔진으로 **기존 RL 모델을 재사용**한다(`engine='rl'`가 기본).
2. 엔진은 **교체 가능 인터페이스(`StowageStrategy`)** 뒤에 둔다 — RL/CP/Greedy 동일 계약(`YardState→CandidatePlan`).
3. 핵심 작업은 "학습"이 아니라 **"통합"** — 원우 모델의 관측/행동 공간을 캐노니컬 스키마에 매핑하는 어댑터.
4. CP-SAT는 **현재 채택하지 않는다.** 동일 인터페이스의 향후 비교군으로만 둔다.

## 근거
- 자산 재사용 + 도메인 전문가 연구 연속성.
- 목표(기술 활용)와 정합 → RL의 연구단계 한계가 채택 차단요인이 아님.
- 학습된 모델이 있으므로 "1달 내 RL 학습" 리스크가 대부분 해소(주 작업이 통합으로 축소).

## 향후 계획 — 조건부 변경(못박음)
- `CPStrategy`(OR-Tools CP-SAT)를 구현하고, **동일 시뮬 인스턴스에서 RL vs CP head-to-head 벤치마크** 실시(04_eval_spec).
- **CP가 유의미하게 우월할 때에만** 기본 엔진을 CP로 교체한다. 그 전까지 기본 엔진은 RL.

## 결과 / 통합 시 확인 리스크
- 원우 모델 입출력 → `CandidatePlan` 어댑팅.
- 추론 의존성·체크포인트·**재현성(시드)** 확보, Gym env 사양(관측/행동/보상) 문서화.
- 시뮬 인스턴스에서 RL 안정성 모니터링(복잡도↑ 저하 리스크) → 불안정 구간은 `GreedyStrategy` 폴백.
- 엔진이 pluggable이므로 향후 비교·교체는 코드 변경 최소.
