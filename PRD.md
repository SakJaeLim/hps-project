# PRD: SNCT 컨테이너 터미널 운영 의사결정 지원 플랫폼

> 범위: **1개월 MVP** — 시뮬레이션 기반 단일 선박 적재계획 의사결정 지원.
> 실시간 스트리밍·본격 RL 재학습·Edge·멀티선석은 Out of Scope. 설계 상세는 `specs/`.
> 본 문서는 AI-Native SDD의 **단일 입력 원천(Single Source of Truth)** 이다. (TASK·TDD가 이 문서를 참조)

## 1. 배경 및 문제 정의
인천신항 SNCT의 적재 계획은 선박 안정성·중량분포·반출순서·재취급 최소화 등 다수 제약을 동시에 고려하는 다단계 의사결정이다.
**핵심 문제**: (1) 숙련 플래너 경험 의존 → 인력 의존·판단 편차, (2) 불필요한 재취급으로 장비·시간 낭비, (3) TOS/AIS/게이트 데이터 분산으로 통합 판단·근거 추적 어려움.

## 2. 목표 (측정 가능)
| 구분 | 목표 |
|------|------|
| 1차(통합) | 선박 1척 요청 → 인식 → 계획 → 검증 → 설명 → 대시보드 전 흐름 작동 |
| 품질 | 제약 위반율 < 1% |
| 성능 | 권고 응답 ≤ 5초 |
| 참고 | 재취급: 기준선(랜덤/단순룰) 대비 비교 (시뮬) |
> 1차 목표는 **기술 통합·작동**이며 성능 경신이 아니다 (ADR-0001).

## 3. 타겟 사용자
주요 페르소나: **적재 계획 플래너 / 운영 관리자**. 한정된 야드·선석에서 빠르고 근거 있는 적재 결정을 내려야 함.

## 4. 사용자 스토리
1. As a 플래너, I want 근거와 함께 적재 권고안을 받기를, so that 판단 편차 없이 빠르게 결정한다.
2. As a 운영관리자, I want 재취급 충돌을 사전에 경보받기를, so that 장비·작업 낭비를 막는다.
3. As a 플래너, I want 자연어로 운영 현황·규정을 질의하기를, so that 근거를 즉시 확인한다. (지식 접근)
4. As a 운영자, I want 네트워크와 무관하게 권고 근거가 사실에 기반하기를, so that 환각 없이 신뢰한다.

## 5. 핵심 기능 (우선순위)
| ID | 기능 | 우선순위 | 관련 spec |
|----|------|---------|-----------|
| F1 | 데이터 어댑터 (시뮬/실데이터, 캐노니컬 스키마) | 상 | specs/03, 06 |
| F2 | 온톨로지(Neo4j) + 제약 검증(Cypher 5종) | 상 | specs/01 |
| F3 | 적재 엔진 (RL 원우 재사용 · Greedy 기준선) | 상 | ADR-0001, specs/00 |
| F4 | 멀티에이전트 흐름 (인식→계획→검증→설명, 재시도) | 상 | specs/02 |
| F5 | 지식 접근 (문서RAG·NL2SQL·GraphRAG) + 설명(xAI) | 중 | specs/07 |
| F6 | 운영자 대시보드 + API | 중 | specs/00 |
| F7 | 평가 하니스 (재취급·위반·속도) | 중 | specs/04 |

## 6. 범위
- **In Scope (1개월)**: F1–F7 의 얇은 종단(E2E). 엔진=RL 통합+Greedy, 지식접근=문서RAG 우선(NL2SQL/GraphRAG 골격).
- **Out of Scope**: 실시간 TOS/AIS 스트리밍, 본격 RL 재학습, Edge INT4, 멀티선석 동시, NL2SQL 자유질의 고도화.

## 7. 기술 스택
Python 단일 런타임 · 질의 Cypher/SQL · 입력 EDIFACT (ADR-0002). LangGraph · Neo4j · PyTorch/SB3 · OR-Tools(향후) · FastAPI · Streamlit.

## 8. 다운스트림 산출물 (SDD 체인)
`PRD.md`(본 문서·SSOT) → `specs/`(설계 상세) → `TASK.md`(원자적 작업) → `tests/`(TDD Red=완료조건) → `src/`(구현=Green). 변경은 항상 상류(PRD/specs) 먼저.
